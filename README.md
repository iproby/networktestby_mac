# mac_net_watch

Консольная утилита для macOS, которая непрерывно следит за интернетом и пытается понять, где именно ломается цепочка:

- сам Mac;
- Wi‑Fi-интерфейс;
- локальный роутер;
- мобильный оператор / uplink;
- DNS;
- VPN / туннель;
- HTTP/TLS-фильтрация.

Теперь у проекта два режима:

- полноэкранный интерфейс в терминале;
- обычный текстовый вывод для логов и отладки.

## Самый простой запуск

Запуск двойным кликом:

- [Запустить диагностику.command](/Users/mac_1/Documents/Codex/Games%20for%20mac/Запустить%20диагностику.command)

Что делает `*.command`:

1. проверяет, есть ли `python3`;

3. запускает красивый интерфейс диагностики.

Если на Mac нет и `python3`, и `Homebrew`, файл остановится и подскажет, что нужно поставить Homebrew.

## Что показывает интерфейс

В полноэкранном режиме видны:

- общий статус `OK / WARN / DOWN`;
- вероятный виновник;
- confidence;
- отдельная гео-таблица доступности сервисов по регионам;
- блок производительности с RX/TX и последним замером Cloudflare edge;
- default route и локальный route;
- интерфейс и IP;
- состояние шлюза;
- внешняя IP-связность;
- системный DNS;
- прямой DNS;
- HTTP-проверки;
- краткий аналитический блок с основаниями и рекомендациями.

Выход из интерфейса: клавиша `q`.

## Что именно проверяет скрипт

Каждый цикл он:

1. читает `default route` и интерфейсы;
2. определяет локальный интерфейс;
3. проверяет TCP-доступность шлюза;
4. проверяет несколько внешних IP без участия DNS;
5. проверяет системный DNS на Mac;
6. делает прямые DNS-запросы к DNS-серверам;
7. открывает контрольные HTTP(S)-URL;
8. отдельно проверяет набор региональных сервисов;
9. периодически делает замер скорости скачивания с Cloudflare edge;
10. формирует вероятную причину проблемы.

## Региональные сервисы по умолчанию

Для более понятной диагностики доступности интернета по странам и регионам скрипт отдельно проверяет:

- Беларусь: `nbrb.by`
- Беларусь: `mvd.gov.by`
- Россия: `yandex.ru`
- Европа: `europa.eu`
- США: `google.com`
- Южная Америка: `gov.br`
- Япония: `yahoo.co.jp`
- Австралия: `abc.net.au`
- Малайзия: `malaysia.gov.my`
- Китай: `baidu.com`

В интерфейсе это показывается отдельным блоком `Гео сервисы`, чтобы было видно, например:

- локально всё живо, но недоступен только Китай;
- Европа и США открываются, а Беларусь и Россия нет;
- все регионы одновременно падают, значит проблема глубже, чем конкретный сайт.

## Основные причины, которые он умеет отличать

- `local_interface_down`
  Проблема на стороне Mac, адаптера, Wi‑Fi-линка или DHCP.

- `gateway_unreachable`
  Mac не может достучаться до роутера. Это сильный сигнал в сторону Wi‑Fi или самого роутера.

- `upstream_wan_failure`
  Роутер доступен, но наружу IP не проходит. Подозрение на мобильного оператора, WAN или внешний апстрим.

- `local_dns_failure`
  Интернет по IP есть, но DNS ломается только на этом Mac.

- `upstream_dns_failure`
  DNS не работает даже напрямую. Это уже больше похоже на проблему выше роутера.

- `tunnel_or_vpn_failure`
  На Mac активен `utun`, и проблема похожа на VPN, туннель или Private Relay.

- `http_layer_failure`
  Базовая сеть жива, а ломается уже HTTP/TLS-уровень.

## Полезные режимы под ваш кейс

Проверить DNS роутера и публичные DNS:

```bash
python3 mac_net_watch.py \
  --dns-server 192.168.5.1 \
  --dns-server 1.1.1.1 \
  --dns-server 8.8.8.8
```

Проверить свои внешние IP:

```bash
python3 mac_net_watch.py \
  --public-endpoint 1.1.1.1:53,443 \
  --public-endpoint 8.8.8.8:53,443 \
  --public-endpoint 185.202.220.144:443
```

Более частый опрос:

```bash
python3 mac_net_watch.py --interval 3 --timeout 1.5
```

Параметры замера скорости:

```bash
python3 mac_net_watch.py \
  --speedtest-interval 120 \
  --speedtest-bytes 5000000 \
  --speedtest-timeout 20
```

Свои региональные сервисы:

```bash
python3 mac_net_watch.py \
  --regional-service "BY|Беларусь|NBRB|https://www.nbrb.by/" \
  --regional-service "BY2|Беларусь|MVD|https://mvd.gov.by/" \
  --regional-service "RU|Россия|Yandex|https://yandex.ru/" \
  --regional-service "CN|Китай|Baidu|https://www.baidu.com/"
```

## Журнал

По умолчанию пишется файл `network-diagnostics.jsonl`.

Туда попадают:

- каждый `sample`;
- `incident_opened`;
- `incident_closed`.

Это удобно, если нужно оставить диагностику на несколько часов, а потом разбирать точные моменты деградации связи.

## Практический сценарий

На Mac клиента запустите:

```bash
python3 mac_net_watch.py --interval 5
```

Или просто откройте:

- [Запустить диагностику.command](/Users/mac_1/Documents/Codex/Games%20for%20mac/Запустить%20диагностику.command)

Оставьте окно работать до очередного обрыва. Когда проблема проявится, интерфейс покажет:

- потерял ли Mac интерфейс;
- доступен ли шлюз;
- есть ли внешняя IP-связность;
- ломается ли только DNS;
- есть ли активный туннель;
- кто сейчас наиболее вероятный виновник.
